================================================================================
  Hallazgos y conclusiones — Ejercicio Serenity BDD (DemoBlaze)
================================================================================

1. Objetivo del ejercicio
   -----------------------
   Automatizar de extremo a extremo el flujo de compra en DemoBlaze usando
   BDD (Cucumber/Gherkin), Serenity BDD y el patrón Screenplay, de forma que
   el escenario sea legible para negocio y mantenible para el equipo técnico.

2. Hallazgos técnicos
   -------------------
   - El flujo cubierto incluye: agregar productos al carrito, validar el
     carrito, completar el formulario de compra y verificar el mensaje de
     confirmación junto con la presencia de un ID de orden.
   - Serenity BDD integra bien Cucumber con reportes vivos (Living
     Documentation) y trazabilidad entre pasos Gherkin, steps Java y evidencias.
   - WebDriverManager evita la gestión manual del ChromeDriver, lo que reduce
     fricción al reproducir el proyecto en otra máquina.
   - El patrón Screenplay (Tasks, Interactions, Questions, Targets) separa
     claramente la intención del actor del detalle de localización y acción
     sobre la UI, lo que facilita cambios en selectores sin tocar el feature.

3. Hallazgos sobre la aplicación bajo prueba (DemoBlaze)
   -------------------------------------------------------
   - Es un sitio demo pensado para prácticas; la disponibilidad y tiempos de
     respuesta dependen de la red, por lo que las pruebas E2E pueden ser
     ocasionalmente sensibles a latencia.
   - Los modales y la navegación por categorías/productos requieren esperas
     explícitas o condiciones de Serenity bien definidas para evitar fallos
     por elementos aún no visibles.

4. Resultado de la última ejecución local de referencia
   -----------------------------------------------------
   - Resumen Serenity: 1 caso de prueba, 1 pasado, 0 fallidos (según
     target/site/serenity/summary.txt tras mvn verify).
   - Failsafe/CucumberTestRunner: 11 tests ejecutados, 0 fallos, 0 errores
     (los steps y hooks cuentan como múltiples pruebas a nivel de runner).

5. Conclusiones
   -------------
   - La combinación BDD + Screenplay + Serenity es adecuada para documentar
     el comportamiento esperado y generar reportes comprensibles para perfiles
     no solo desarrollo.
   - Para una entrega académica o de revisión externa, conviene versionar
     instrucciones claras (readme.txt), scripts de ejecución y, si se pide,
     una copia de los reportes HTML en carpeta dedicada tras una corrida
     exitosa, además del código fuente y el pom.xml.
   - Las pruebas E2E sobre aplicaciones web públicas deben considerarse
     dependientes del entorno; la reproducibilidad técnica del proyecto es
     alta (Maven + Java + Chrome), pero el resultado puntual puede variar si
     el sitio cambia o presenta lentitud.

6. Posibles mejoras futuras
   -------------------------
   - Añadir más escenarios (datos inválidos, carrito vacío, cancelación).
   - Parametrizar datos sensibles vía variables de entorno o ficheros externos
     no versionados si en el futuro se usaran credenciales reales.
   - Integrar la suite en CI (por ejemplo GitHub Actions) para ejecución
     programada con artefactos de reporte publicados.
